% ─── Abstract ────────────────────────────────────────────────────────────────
\begin{abstract}
\noindent
AI agents deployed in enterprise Atlassian environments fail in ways that are fundamentally non-reproducible using classical debugging methods. When an agent re-executes, the LLM produces different outputs; worse, re-execution may trigger side-effecting tools — sending emails, modifying tickets — during the debug session itself. \textbf{gommage} is an infrastructure-layer proxy that (i) transparently records the full trajectory of any LLM agent run as a structured \textit{Agent Execution Record} (AER), (ii) replays that trajectory deterministically by intercepting and mocking all tool calls, (iii) surfaces this workflow natively inside Jira, and (iv) employs a transport-layer verification mechanism intercepting calls at the stdio, shell, and filesystem layers to reconcile self-reported AER fields. This document justifies the technical choices underpinning gommage, grounds them in the AER provenance model of Vispute (\numberfont 2026\rmfamily), and presents the \numberfont 5\rmfamily-day implementation plan.
\end{abstract}

% ─── 1. Problem ──────────────────────────────────────────────────────────────
\section{Problem Framing}

\subsection{The Debugging Gap}

Classical software debugging relies on two guarantees: \textit{reproducibility} (running the same code on the same input produces the same state) and \textit{isolation} (reproducing a bug does not cause additional side effects in production). Both guarantees collapse for LLM-based agents.

\begin{itemize}[leftmargin=1.5em, itemsep=0.3em]
  \item \textbf{Non-reproducibility:} The same instruction passed to an LLM agent twice yields different tool call sequences, reasoning chains, and outputs due to sampling stochasticity. Re-running a failed agent to debug it produces a different failure — or no failure at all.
  \item \textbf{Side effects during debug:} A failed Jira triaging agent, re-run by an engineer, may send a customer email or overwrite a ticket during the debug session — compounding the original damage.
  \item \textbf{Opaque state:} Simple log files capture what happened but not \textit{why}: what prompt the LLM received at each step, what context was in its window, and what it inferred before each action.
\end{itemize}

What is missing is an enterprise-grade \textbf{Flight Recorder}: infrastructure that captures every LLM input, context variable, and tool-call payload during a live run, then allows developers to re-execute the agent step-by-step using the exact recorded tool responses — guaranteeing determinism without touching live APIs.

\subsection{Scope and Target System}
gommage targets agents deployed in the Atlassian ecosystem (Jira Software, Jira Service Management) that interact with external tools (databases, email, webhooks). The demo agent is a \textbf{Jira ticket triaging agent} that reads incoming tickets, queries a database for SLA context, and routes tickets to teams — with an email notification on completion.

% ─── 2. Methodology ──────────────────────────────────────────────────────────
\section{Theoretical Foundation}

\subsection{The Agent Execution Record (AER)}

Our core data model is grounded in Vispute (\numberfont 2026\rmfamily), \textit{Reasoning Provenance for Autonomous AI Agents} \cite{vispute2026}. Vispute formalises a distinction that is central to our design:

\begin{quote}
\dialogfont\large\itshape
"State checkpoint systems save serialized snapshots of the agent's computational state. What they do not contain as a structured field is \textbf{why} the agent chose each action, \textbf{what it concluded} from each observation, or \textbf{how each conclusion shaped its strategy}."
\end{quote}

The AER model introduces structured reasoning provenance as a first-class schema primitive. Vispute further defines three replay modes — \textbf{mock replay} being the one gommage implements — which re-executes the agent using recorded tool responses without calling live endpoints.

We extend the AER concept in two directions suited to our Jira-native context:
\begin{enumerate}[leftmargin=1.5em, itemsep=0.3em]
  \item \textbf{Side-effect classification field:} Each tool call record is tagged with a \texttt{side\_effect} boolean and a \texttt{mock\_required} flag, enabling the Mock Registry to automatically intercept destructive calls during replay.
  \item \textbf{Jira attachment envelope:} The AER is serialised with a top-level \texttt{jira\_ticket\_id} field and pushed as a structured attachment to the originating ticket via the Jira REST API.
\end{enumerate}

\subsection{Pipeline Mechanics}
During a **live run**, the unique \texttt{run\_id} binds to the Jira ticket. Every call to the Anthropic API is captured as an \texttt{LLMStep}, and every tool execution is captured as a \texttt{ToolStep}. Upon completion, the AER Serialiser assembles these into a JSON structured trace attached natively to Jira.

During a **replay run**, the developer triggers execution from the Jira Replay Panel. The Mock Registry pre-loads all tools tagged \texttt{side\_effect: true}. As the agent walks the graph, the Tool Proxy intercepts external calls, queries the Mock Registry, and safely returns the exact output the agent saw originally.

\subsection{AER Schema Definition}
To guarantee Pydantic v\numberfont 2 \rmfamily validation and robust typing, gommage utilizes the following structure:

\begin{lstlisting}[language=Python]
class AERStep(BaseModel):
    step_id: int
    step_type: Literal["llm_call", "tool_call"]
    
    # LLM Context
    messages: Optional[List[dict]]      
    response: Optional[str]             
    
    # Tool Context
    tool_name: Optional[str]
    tool_input: Optional[dict]
    tool_output: Optional[Any]
    side_effect: bool = False           # Flag for destructive calls
    mock_required: bool = False         # Enforces mock on replay
    
    # Reasoning Provenance (Vispute 2026)
    intent: Optional[str]              
    inference: Optional[str]           
    evidence_chain: Optional[list]     

class AgentExecutionRecord(BaseModel):
    run_id: str
    jira_ticket_id: str
    steps: List[AERStep]
    status: Literal["success", "failure", "partial"]
\end{lstlisting}

% ─── 4. Technology Choices ───────────────────────────────────────────────────
\section{Technology Choices \& Justification}

\begin{itemize}[leftmargin=1.5em, itemsep=0.5em]
  \item \textbf{LangGraph \& LLM API:} LangGraph's checkpointer provides natural hooking points for graph super-steps, whilst LLM Sonnet excels at structured output generation and maintains high fidelity across long context arrays during deterministic replay.
  \item \textbf{Atlassian Forge (UI Kit \numberfont 2\rmfamily):} The Jira-native replay panel runs serverlessly inside Atlassian's infrastructure. Forge's \texttt{@forge/api} allows direct access to the AER attachment, ensuring sensitive trace data never leaves the customer's tenant boundary.
  \item \textbf{SQLite (Demo) to PostgreSQL (Prod):} Local development leverages SQLite for immediate portability. Production scale-out utilizes PostgreSQL to facilitate cross-run behavioral analytics \cite{vispute2026}.
\end{itemize}

% ─── 5. Feature Design ───────────────────────────────────────────────────────
\section{Feature Design \& Divergence Tracking}

\subsection{Mock Registry \& Safety Guards}
The Mock Registry prevents live tool execution during replay. Detection uses both static patterns (\texttt{send\_email}, \texttt{execute\_sql\_write}) and dynamic AER-driven classification (HTTP \numberfont 201/202 \rmfamily status codes observed during the live run). A \textbf{MockBadge} UI component renders inside Jira so developers explicitly see which API calls were suppressed.

\subsection{Divergence Editing}
The true power of gommage lies in Divergence Editing. A developer can modify the prompt at step $N$. The LLM produces a new output, branching the trajectory. From $N+1$ onward, gommage continues to supply recorded tool responses where applicable, but the agent's logic pathway updates.

% ─── 6. Evaluation ───────────────────────────────────────────────────────────
\section{Evaluation Methodology}

We define two primary metrics to assess gommage's correctness during the hackathon testing phase.

\begin{tcolorbox}[metricbox]
  \textbf{\titlefont\large Replay Fidelity Score (RFS)} — measures whether replay re-presents identical contexts.
  \begin{center}
    {\numberfont\Large RFS $\geq$ 0.95}
  \end{center}
  \textit{Target: RFS $\geq \numberfont 0.95$ \dialogfont on synthetic test suite.}
\end{tcolorbox}

\vspace{0.2em}

\begin{tcolorbox}[metricbox]
  \textbf{\titlefont\large Mock Recall Rate (MRR)} — measures whether all side-effecting tool calls are intercepted.
  \begin{center}
    {\numberfont\Large MRR = 1.0}
  \end{center}
  \textit{Target: MRR = \numberfont 1.0 \dialogfont (Zero false negatives — absolutely no live calls during replay).}
\end{tcolorbox}

% ─── 8. Conclusion ───────────────────────────────────────────────────────────
\section{Conclusion}

gommage addresses a structural gap in enterprise AI agent infrastructure: the inability to reproduce, inspect, and safely debug LLM agent failures in production. By grounding the system in the AER provenance model \cite{vispute2026}, building a transparent proxy that requires no modifications to the agent's core architecture, and surfacing the entire debugging workflow natively inside Jira, we deliver a system where AI is the mechanism — not a feature. Remove the structured recording, deterministic replay, and side-effect mock layer, and the debugging problem is unsolved. That is the bar the specification sets, and that is the bar gommage is built to clear.
